1. EXECUTIVE SUMMARY
1.1 Product Vision
Mission Statement:"Make quality food accessible and affordable while reducing food waste in India's tier-2 cities."
Product Description:A mobile marketplace connecting restaurants with surplus fresh food to customers looking for affordable meals. Customers purchase "Surprise Boxes" at 60-70% discount without knowing exact contents, pickup same evening.
Key Differentiators:
WhatsApp-first MVP, then native app
Pickup-only (no delivery overhead)
Mystery box gamification
Hyperlocal focus (Hubli-first)
Lower commission than competitors (20-30% vs 35-40%)

1.2 Success Metrics
Customer Metrics:
Monthly Active Users (MAU): Target 5,000 by Month 6
Repeat purchase rate: >40%
Average orders per user per month: 2-3
Customer acquisition cost: <₹50
Net Promoter Score: >50
Restaurant Metrics:
Active restaurant partners: 50 by Month 6
Average boxes per restaurant per day: 8-10
Restaurant retention rate: >80%
Average revenue per restaurant: ₹20,000/month
Business Metrics:
Gross Merchandise Value (GMV): ₹10L by Month 6
Take rate: 25-30%
Monthly revenue: ₹2.5L-3L by Month 6
Unit economics: Positive by Month 3

3.4 Edge Case Flows
3.4.1 Customer Cancellation Flow
Customer wants to cancel reservation
│
├─> Opens "My Reservations"
├─> Taps on active reservation
├─> "Cancel Reservation" button
│
├─> If >2 hours before pickup:
│   ├─> "Are you sure?" confirmation
│   ├─> User confirms
│   ├─> Reservation cancelled
│   ├─> Box becomes available again
│   ├─> Restaurant notified
│   └─> No penalty
│
├─> If <2 hours before pickup:
│   ├─> Warning: "Cancellation fee ₹20 applies"
│   ├─> "Still cancel?"
│   ├─> If yes:
│   │   ├─> ₹20 deducted from wallet/next purchase
│   │   ├─> Reservation cancelled
│   │   └─> Restaurant compensated
│   └─> If no:
│       └─> Returns to reservation details
│
└─> If customer no-shows:
├─> Restaurant marks "No-show" after 30 min grace period
├─> Customer account flagged
├─> After 3 no-shows:
    │   └─> Requires ₹50 advance deposit for future reservations
    └─> Can appeal/explain via support

END

3.4.2 Restaurant Out-of-Stock Flow
All boxes sold out earlier than expected

├─> Restaurant taps "Mark as Sold Out"

├─> Confirmation:

4.1.8 Payment Handling (Cash-only MVP)
Customer: Pays cash at pickup (₹99)
Restaurant: Receives full amount, pays commission to platform weekly
Platform: Collects commission via bank transfer/UPI from restaurant
Transparency: Restaurant sees commission deduction in real-time
Commission Structure:
Standard: 30%
Early partners (first 20): 20%
High-volume (50+ boxes/month): 25%
Acceptance Criteria:
Customer sees "Pay cash at pickup" clearly
Restaurant receives payment breakdown
Weekly payment run every Friday
Invoice/receipt generated automatically
Dispute resolution process defined

6.1 Technology Stack
Frontend (Mobile Apps)
Customer App:
Framework: React Native (cross-platform iOS & Android) 
Why: Single codebase, faster development, 90% code sharing
Alternative: Flutter (if team prefers Dart)
State Management: Redux Toolkit + Redux Persist
Navigation: React Navigation v6
UI Components: React Native Paper (Material Design)
Maps: react-native-maps (Google Maps integration)
QR Code: react-native-qrcode-scanner
Notifications: Firebase Cloud Messaging (FCM)
Analytics: Firebase Analytics + Mixpanel
Crash Reporting: Firebase Crashlytics
Restaurant Partner App:
Same stack as customer app (shared components where possible)
Additional: react-native-camera (QR scanning)
Why React Native over Native:
Faster time-to-market (6-8 weeks vs 12-16 weeks)
Easier to find developers
Lower maintenance cost (one codebase)
Performance adequate for this use case (not gaming/heavy graphics)

Backend
API Server:
Framework: Node.js + Express.js 
Why: JavaScript everywhere (frontend devs can contribute)
Fast development, huge ecosystem
Alternative: Python + FastAPI (if team prefers Python)
Database: PostgreSQL (primary) 
Relational data (users, restaurants, boxes, reservations)
ACID compliance important for financial transactions
Cache: Redis 
Session storage
Real-time availability caching
Rate limiting
File Storage: AWS S3 / Google Cloud Storage 
Restaurant photos
User uploaded photos (ratings)
QR codes
Search: Elasticsearch (later phase) 
Restaurant search
Location-based queries
Real-Time Features:
WebSocket: Socket.io 
Real-time availability updates
Reservation notifications
Chat (if we add customer-restaurant chat)
Background Jobs:
Queue: Bull (Redis-based) 
Send notifications (async)
Generate daily reports
Process payments
Authentication:
JWT Tokens (JSON Web Tokens) 
Stateless authentication
30-day expiry

Payments: (Phase 2)

UPI, cards, wallets
Escrow handling

Infrastructure

6.3 Data Flow Examples
Example 1: Customer Reserves a Box
1. User taps "Reserve" button
   │
├─> App sends POST request:
   │   POST /api/reservations
   │   Body: {boxId: 123, userId: 456}
   │   Headers: {Authorization: "Bearer JWT_TOKEN"}
   │
2. API Gateway → Auth Service
   │
├─> Verify JWT token
├─> If invalid → Return 401 Unauthorized
   └─> If valid → Proceed
   │
3. API Server receives request
   │
├─> Check box availability (query Redis cache)
   │   └─> If sold out → Return 409 Conflict
   │
├─> Create reservation record (PostgreSQL)
   │   INSERT INTO reservations (box_id, user_id, status, ...)
   │
├─> Decrement box availability (Redis + PostgreSQL)
   │   UPDATE boxes SET quantity = quantity - 1 WHERE id = 123
   │
├─> Generate reservation code (e.g., SB-1234)
   │
├─> Generate QR code (encode {reservationId, userId, amount})
   │   └─> Save QR image to S3
   │
4. Add jobs to queue (async)
   │
├─> Send WhatsApp notification to customer
├─> Send push notification to restaurant
├─> Send email confirmation (if email provided)
   │
5. Return success response to app
   │
   └─> {
        "success": true,
        "reservation": {
          "id": 789,
          "code": "SB-1234",
          "qrCodeUrl": "https://s3.../qr-code.png",
          "pickupTime": "8:00-9:00 PM",
          "restaurant": {...}
        }
      }
   │
6. App navigates to Confirmation screen
   │
7. Background workers process queue
   │
├─> Twilio API: Send WhatsApp message
├─> Firebase FCM: Send push notification
   └─> SendGrid: Send email
   │
8. WebSocket broadcasts to all connected clients
   │
   └─> "Box 123 availability updated: 2 left"
   │
9. Restaurant app receives notification
   │
   └─> Push notification: "New reservation! SB-1234"

DONE

Example 2: Restaurant Lists Boxes
1. Restaurant owner taps "List Boxes"
   │
├─> App sends POST request:
   │   POST /api/listings
   │   Body: {
   │     restaurantId: 789,
   │     quantity: 5,
   │     price: 99,
   │     pickupStart: "20:00",
   │     pickupEnd: "21:00",
   │     description: "Pizza slices, garlic bread..."
   │   }
   │
2. API Server receives request
   │
├─> Validate input (quantity 1-20, price 80-120, etc.)
   │
├─> Create listing record (PostgreSQL)
   │   INSERT INTO listings (restaurant_id, quantity, price, ...)
   │
├─> Cache in Redis for fast retrieval
   │   SET listing:{listingId} {JSON}
   │   EXPIRE listing:{listingId} 21600  (6 hours TTL)
   │
├─> Add to geospatial index (Redis)
   │   GEOADD listings {longitude} {latitude} {listingId}
   │
3. Return success response
   │
4. Add job to queue
   │
├─> Send notifications to nearby customers
   │   └─> "New boxes available near you!"
   │
5. WebSocket broadcasts to all connected customers
   │
   └─> "New listing available: Pizza Corner"
   │
6. Customer apps refresh home feed
   │
   └─> Query API: GET /api/boxes?lat=X&lon=Y&radius=5
   │
7. API responds with listings
   │
├─> Query Redis geospatial index
   │   GEORADIUS listings {lon} {lat} 5 km
   │
├─> Fetch listing details from cache/DB
   │
   └─> Return JSON array of boxes

```
## 7.2 Redis Data Structures

**Purpose:** Fast caching, real-time availability, session storage

### Key Patterns:
```
1. Listing Availability (String)
   Key: listing:{listingId}:available
   Value: 2 (quantity available)
   TTL: 21600 sec (6 hours)

   Example:
   SET listing:123:available 2
   EXPIRE listing:123:available 21600

2. Geospatial Index (Geo)
   Key: listings:geo
   Members: {listingId}
   Coordinates: {longitude, latitude}

   Example:
   GEOADD listings:geo 75.1240 15.3647 123
   GEORADIUS listings:geo 75.1240 15.3647 5 km WITHDIST

3. User Sessions (Hash)
   Key: session:{userId}
   Fields: {token, fcmToken, lastActive, etc.}
   TTL: 2592000 sec (30 days)

   Example:
   HSET session:456 token "jwt_token_abc"
   HSET session:456 lastActive "2026-03-05T16:00:00Z"
   EXPIRE session:456 2592000

4. Real-time Reservations Counter (String)
   Key: listing:{listingId}:reserved_count
   Value: 3 (number of reservations)

   Example:
   INCR listing:123:reserved_count
   GET listing:123:reserved_count

5. Rate Limiting (String)
   Key: ratelimit:{userId}:{endpoint}
   Value: 5 (requests made)
   TTL: 60 sec

   Example:
   INCR ratelimit:456:POST_reservations
   EXPIRE ratelimit:456:POST_reservations 60
   GET ratelimit:456:POST_reservations

## 8.2 WebSocket Events

**Connection:** `wss://api.surprisebox.in/ws`

**Authentication:** Send JWT token on connection

### Client → Server Events

#### `subscribe_listings`

Subscribe to listing updates in a specific area.

**Payload:**
```json
{
  "event": "subscribe_listings",
  "data": {
    "latitude": 15.3647,
    "longitude": 75.1240,
    "radius": 5
  }
}
```

#### `subscribe_reservation`

Subscribe to updates for a specific reservation.

**Payload:**
```json
{
  "event": "subscribe_reservation",
  "data": {
    "reservationId": 1001
  }
}
```

---

### Server → Client Events

#### `listing_updated`

Sent when a listing's availability changes.

**Payload:**
```json
{
  "event": "listing_updated",
  "data": {
    "listingId": 123,
    "quantityAvailable": 2,
    "updatedAt": "2026-03-05T16:30:00Z"
  }
}
```

#### `listing_sold_out`

Sent when a listing sells out.

**Payload:**
```json
{
  "event": "listing_sold_out",
  "data": {
    "listingId": 123,
    "soldOutAt": "2026-03-05T17:00:00Z"
  }
}
```

#### `new_reservation`

Sent to restaurant when new reservation is made.

**Payload:**
```json
{
  "event": "new_reservation",
  "data": {
    "reservationId": 1001,
    "code": "SB-1234",
    "customerName": "Priya M.",
    "expectedTime": "20:15:00"
  }
}
```

#### `reservation_cancelled`

Sent to restaurant when customer cancels.

**Payload:**
```json
{
  "event": "reservation_cancelled",
  "data": {
    "reservationId": 1001,
    "code": "SB-1234",
    "cancelledAt": "2026-03-05T17:00:00Z"
  }
}
```
